## 2024-05-23 - [Preventing Unbounded Memory Allocation DoS in Network Readers]
**Vulnerability:** A Denial of Service (DoS) vulnerability existed in the TCP listener (`x32_tcp/src/main.rs`). The server read incoming data until a newline character was found using `reader.read_line(&mut line)`. A malicious client could send an infinite stream of bytes without a newline, causing the server to allocate memory until it crashed with an Out Of Memory (OOM) error.
**Learning:** This is a common pattern when reading from untrusted sources in Rust. `BufRead::read_line` and similar functions (like `read_to_end` or `read_to_string`) are dangerous when used on network streams without length limits, because they trust the sender to provide the expected delimiter or EOF before consuming excessive memory.
**Prevention:** Always bound reads from untrusted sources. Use the `std::io::Read::take(limit)` adapter to wrap the reader before calling unbounded reading functions. For example: `reader.by_ref().take(4096).read_line(&mut line)`. This ensures that even if a delimiter is never sent, the allocation is strictly bounded to the specified limit.
## 2024-05-24 - [DoS via Unbounded Line Reads]
**Vulnerability:** The `x32_tcp` application used `BufRead::read_line` to accept incoming commands without a size limit. An attacker could establish a connection and send a continuous stream of bytes without a newline character, causing the application to continuously allocate memory.
**Learning:** Rust's `BufRead::read_line` will read until it encounters a newline or EOF. If an input source is untrusted, this can lead to unbounded memory allocation and eventual Out Of Memory (OOM) crashes.
**Prevention:** When reading lines from untrusted sources (e.g., network connections), always limit the maximum number of bytes read. This can be achieved efficiently using the `take()` adapter on the reader (e.g., `reader.by_ref().take(LIMIT).read_line(&mut buf)`).

## 2024-03-21 - [DoS via Unbounded read_line/lines() from STDIN]
**Vulnerability:** Several CLI tools (`x32_get_scene`, `x32_set_scene`) used unbounded `io::stdin().read_line()` or `io::stdin().lock().lines()` to read piped inputs or manual inputs. This can lead to unbounded memory allocation and Out-Of-Memory (OOM) crashes if a script or an attacker pipes a massive stream of data without a newline.
**Learning:** The same vulnerability pattern identified in networked tools (like `x32_tcp`) also applies to local CLI tools processing piped input. An unbounded `read_line` or `.lines()` iterator continuously allocates memory until it finds a `\n` or EOF.
**Prevention:** When reading lines from any untrusted or uncontrolled source (including STDIN, which can be piped), limit the maximum number of bytes read using the `take()` adapter on the reader (e.g. `stdin_lock.by_ref().take(4096).read_line(&mut buf)`).

## 2024-06-12 - [DoS via Panic in OSC Blob Parsing]
**Vulnerability:** The `x32_automix` application used `try_into().unwrap()` when parsing meter data from OSC blobs. A malformed or truncated OSC message from the mixer (or an attacker spoofing the mixer) would cause the application to panic and crash.
**Learning:** Using `unwrap()` on data derived from external network sources is a security risk. Even if the protocol specifies a certain length, network data should always be treated as untrusted and potentially malformed.
**Prevention:** Always use safe parsing methods like `.get()` for slicing and `try_into().ok()` or `match` for conversions from network data. Ensure all indexing into fixed-size state vectors is bounds-checked or uses safe accessors like `.get()` and `.get_mut()`.

## 2024-06-21 - [DoS via Unbounded read_line from STDIN in Replay Tool]
**Vulnerability:** The `x32_replay` tool used unbounded `stdin.read_line(&mut line)` inside an endless loop to process user commands. An attacker could pipe a massive stream of characters without a newline, causing the application to endlessly allocate memory until it crashed with an Out-Of-Memory (OOM) error. Furthermore, without checking for empty lines upon EOF, the loop would rapidly spin on `Ok(0)`, causing a 100% CPU lockup Denial of Service.
**Learning:** Even local interactive tools must safely bound their standard input reads. An unbounded `read_line` in an interactive loop is vulnerable to piped payloads. Additionally, wrapping a reader with `.take()` requires explicitly checking for EOF (e.g. `line.is_empty()`), otherwise `read_line` will silently return `Ok(0)` on every subsequent iteration, causing CPU exhaustion if not explicitly `break`'d.
**Prevention:** Bound STDIN reads using `take(LIMIT)`: `stdin_lock.by_ref().take(4096).read_line(&mut line)`. Always explicitly verify the line is not empty (or `len == 0`) to detect EOF and break the loop to prevent CPU exhaustion.
## 2024-06-25 - [DoS via Panic in Numeric OSC Seed Parsing]
**Vulnerability:** The `x32_core` library (which drives the emulator and other state-seeding contexts) used `arg_value.parse().unwrap()` when parsing integer ("i") and float ("f") values from OSC seed lines (e.g., in `seed_from_lines`). A malformed or unexpectedly formatted numeric string in a seed file would cause the application to panic and crash.
**Learning:** Hardcoded configuration files or external seed inputs can sometimes contain invalid data due to user error or tampering. Using `.unwrap()` on string parsing functions (`.parse()`) assumes the data is always perfectly formatted, which violates the principle of not trusting external inputs.
**Prevention:** In state-seeding and configuration parsing contexts, use safe parsing constructs like `if let Ok(val) = arg_value.parse()` or `match`. If parsing fails, gracefully handle the error (e.g., by skipping the line with `continue` and potentially logging a warning) instead of allowing the application to panic.
